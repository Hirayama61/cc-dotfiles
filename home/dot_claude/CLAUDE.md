# ~/.claude/CLAUDE.md — グローバル作業規約

全プロジェクト共通。
hook で守れる事項は書かない。
ここには hook 化できない判断の核のみ。

## 出力スタイル

- 一文ごとに改行する。
- 構造化は markdown 見出し(##/###)で行う。太字はセクション見出し的な要点に絞り、本文中で乱発しない。
- 箇条書きは薄い羅列を避ける。各項目は 1-2 文の中身を持たせ、単純な話は散文で返す。
  ファイル一覧・手順・差分提示など箇条書きが読みやすい場面はこの限りでない。
- 単純な質問には短く答え、不要に膨らませない。
- 雰囲気語や造語を使わない。専門語はかみ砕く。
- 提案や変更は diff で見せる。

## 対話の作法

- 誤りは認めるが、過剰謝罪・自己卑下に崩れず問題そのものに集中する。
- 質問は一度に1つまで。曖昧でもまず暫定回答を出してから確認する。

## オーケストレーション(コンテキスト基準の分業)

メインが直接実行するのが既定。
委譲は賢さの節約ではなく、メインのコンテキスト保護のために使う。

- **メイン直接が既定**: 設計判断・実装・デバッグの中核はメインが自分でやる。
  会話と判断の文脈を手元に保ったまま進めるのが最速で品質も高い。
- **委譲する基準**: 次のいずれかに当たる時だけ `Agent` に1タスクずつ委譲する(委譲先は下記「委譲先の使い分け」に従う)。
  (1) 大量出力を伴う探索・ログ漁り・長大ファイル読み(メインに必要なのは結論だけ)。
  (2) 互いに独立で並列化できる作業(`run_in_background`)。
  (3) 並行して進めたい変更作業(`isolation: "worktree"`)。
  (4) コンテキスト隔離が目的の独立検証・反証レビュー(会話の文脈や実装意図を渡さず判断させる)。
- **委譲先の使い分け(コスト最適化)**: 実装・設計の中核と独立検証・反証レビュー(基準4)は `delegate`(model: inherit、親と同格)。
  検索・読込だけなら `scout`(haiku 固定)、Web/ドキュメントの定型調査は `researcher`(sonnet 固定)、
  仕様確定済みの機械的編集・定型実装は `worker`(sonnet 固定)。
  専用レビュアー(code-reviewer / security-reviewer / design-reviewer)は各 skill の定義どおり起動する。
  ティア固定は各エージェント定義側にあるため、メインのモデルに依らず同じ定義で動く。
  実装・設計の中核を委譲する時はモデルを下げない(`worker` に設計判断をさせない)。
- **委譲前の外部脳検索(種渡し)**: 関連知見が判断に影響しうる委譲では、事前に `~/obsidian/brain` を keyword 検索し、関連ノートを「既知の前提」として委譲先に渡す。
- **プランレビューゲート**: 委譲した破壊的・外向き・広範囲・業務判断を含むタスクは先出しプランを軽量レビュー(再立案はしない)。
  それ以外の委譲は一段で実行してよい。
- **人間へのエスカレーション**: 判断・曖昧点・破壊的/外向き操作は、委譲の有無に依らず作業前に人間へ確認する。
- **計画**: 複数ステップ依頼はメインが Plan モードを通してから着手する。
- **タスク追跡はネイティブ Task ツール**: エージェント内部の一時的な進捗管理ファイル(TODO 一覧・進捗メモ)を自前で作らない(ユーザーが明示依頼した文書は対象外)。
  外部脳の成果物ファイル(plan/result.md 等)を進捗チェックリストとして使わない(用途は delegate.md の定義どおり)。
- **完了通知**: background 完了イベントで判断やりとりを中断しない。
  安全な切れ目で reconcile する。
- **外部脳への reconcile**: 委譲先から `外部脳候補` / 鮮度 flag を受けたら安全な切れ目で reconcile する。
  obsidian-memory で書き戻し、陳腐化は callout 追記 or 申告。
- **判断の記録**: 重要な二者択一は AskUserQuestion を積極利用。
  確定後に obsidian-memory で理由・文脈付きの Decisions ノートを書く。
  timing とファイル分割はメイン判断。
- **学びはガイドへ(guide-capture)**: 運用知(テスト規約・実装の注意点・落とし穴)を学んだら guide-capture skill での反映を提案する(人間ゲート)。
  ガイドは現在状態のみ、「なぜ=履歴」は Decisions へ。
- **doc-gravity**: Claude が生成する作業ドキュメントは `~/obsidian/brain/Tasks/<repo>/` に書く(`<repo>` は `resolve-repo-key.sh` で導出)。
- **grill-with-docs の Obsidian 振替**: 用語集は `~/obsidian/brain/Tasks/<repo>/CONTEXT.md`、ADR は Decisions ノートへ。
  セッション開始時の走査は CONTEXT.md の1パスのみ。

## worktree 並行作業(1 worktree = 1 ブランチ)

判断軸は「案件」ではなく**ブランチ**。
人間も Claude も同一規約に従う。

- **並行作業はディスパッチャ型**: 並行させる時はメインが worktree を用意し、実作業を委譲先エージェント(使い分けはオーケストレーション節)/ `gwq exec` / `git -C <path>` に振る。
  単独作業はオーケストレーション節どおりメイン直接でよい。
- **戻しにくい変更は worktree で**: 履歴改変・大量書換など影響の大きい変更は、単独作業でも worktree に隔離してから行う。
- **worktree 作成は `~/ghq/github.com/Hirayama61/dotfiles/bin/wt.sh` 経由のみ**: どのリポからもこの絶対パスで呼ぶ。
  これは Bash で直接 worktree を作る操作(git worktree add / gwq add / claude --worktree)への規約。
  `isolation: "worktree"`(Agent ツール経由)は WorktreeCreate hook が同じ flat 配置へ誘導するため、自粛せず使ってよい。
  hook が使えない環境(jq / dotfiles リポ / origin 不在)では既定の `<repo>/.claude/worktrees/` 配置に戻る。
  worktree 隔離は混線防止であり権限境界ではない。
- **1 worktree = 1 ブランチ**: 同じブランチの続きは既存 worktree を使う。
  新規作業前にこの分岐をまず判断する。
- **案件 = Epic 配下の複数タスク**: 複数ブランチが要るなら epic ブランチを派生させタスクごとの worktree を並べる。

## 調査と実装の規律

- **未検証を断言しない**: 確信のない点はヘッジするか flag する。
- **裏取りはコストで判断する**: コマンド数秒で済む確認は自由にやる。
  時間・トークンを大きく使う大規模な裏取りは、着手前にユーザーへ許可を取る。
- **決定の蒸し返しをしない**: ユーザーが既に下した判断・好みは裏取りで正当化しない。
- **跨ぎ前提は再検証**: セッション/時間を跨ぐ前提は依拠する前に都度確認する。
- **既存パターンに合わせる**: 実装前に同種ファイルを複数調べ prevailing なパターンを把握してから沿う。
- **詰まったら codex-consult**: 同一問題で3回試しても解決しない時は別モデルのセカンドオピニオンを取る。
