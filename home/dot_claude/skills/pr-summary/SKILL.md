# pr-summary

Claude が生成する **PR 概要(PR body)** を、レビュアーが **1〜2分で本質を把握**
できる構成にする。北極星は **「レビュアーの負荷を下げ / 必要な情報だけ書く」**。
対象は **PR 概要のみ**(Plan / Obsidian ノートは対象外)。

このスキルは「運用しながら育てる」前提。規約本体(下記の節)は後から差し替え
やすいよう findings の節構成に揃えている。

## 設計の核(なぜこの形か)

- **BLUF(Bottom Line Up Front)**: 結論・主旨を冒頭に置く。読者が本文を漕がずに
  最重要情報へ即着地する。「まず全体像、次に必要なら詳細」という PR レビューの
  読み方に直結する。
- **逆ピラミッド**: 重要度の高い順に上から並べる。途中で読むのをやめても要点は
  取れている状態。
- **Progressive disclosure(段階開示)**: 詳細は `<details>` 折りたたみに退避し、
  デフォルト表示を要点だけに保つ。GitHub は PR body 内の `<details><summary>` を
  ネイティブにレンダリングするので、長いログ・全ファイルリスト・設計の深掘りは
  ここに畳む。
- **差分の再記述をしない**: PR body は「WHAT が変わったか・WHY(動機)・どう検証
  したか」を高レベルで語る。HOW(コードがどう動くか)は diff とインラインコメントが
  語る領域で、body で繰り返さない。レビュアの最大コストは「文脈の不在で diff を
  読みながら推測すること」なので、body の仕事は **diff を読む前の地図を渡すこと**。

> レビュー効果は 200〜400 行を超えると急落し、+100 行ごとに review 時間が倍増する。
> body の速読性は「PR を小さく保つ」とセットで効く。

## 構造規約(PR body が満たすべき規範)

### 必須(順序固定 = 逆ピラミッド)

1. **1行サマリ(結論先出し)**: 「何を・なぜ」を1文で。タイトルの言い換えではなく、
   レビュアが最初に知るべき結論。例「セッション cleanup の競合を直し、高負荷時の
   502 を解消」。
2. **なぜ(動機/背景)**: 2〜4行。何の問題を解決するか。関連 issue があれば
   `Closes #123`。background が自明なら省略可。
3. **主な変更**: 概念単位で 3〜7 項目の箇条書き。ファイル名の羅列ではなく
   「何をしたか」を概念でまとめる。各項目1行。
4. **検証**: どう動作確認したか(テスト追加・手動手順・スクショ)。「未検証」なら
   正直にそう書く。

### 任意(必要時のみ・折りたたみ推奨)

- **レビュー観点 / 注意点**: 重点的に見てほしい箇所、トレードオフ、未解決の判断点。
  視線誘導が効く。
- **詳細(`<details>` に畳む)**: 長いログ、全ファイルリスト、設計の深掘り、代替案の
  検討。

### 「1〜2分で読める」ための上限目安

- **冒頭3行以内で結論が立つ**(1行サマリ + 動機の頭)。スクロールせず要点が取れる。
- **デフォルト表示は本文 ~600〜800 字(日本語)/ ~120 行以内** を上限の目安に。
  超える分は `<details>` へ退避。
- **箇条書きは1ブロック最大7項目**。それ以上は概念で束ねるか折りたたむ。
- **見出しは最大4つ**(動機/変更/検証/+1)。見出し乱立は走査を妨げる。
- **小規模 PR は見出しゼロ + 数行**で十分(後述の Compact テンプレ)。

## テンプレ(2段階・既定は Compact 倒し)

PR 規模で2段階。**「迷ったら Compact」** を既定にする(オーバーヘッドの最小化が
速読の本質)。判定は「変更が **1〜2 概念**に収まり、レビュー観点の指示が不要なら
**Compact**、そうでなければ **Standard**」。デフォルトを Compact に倒す。

### 既存テンプレ・トレーラの扱い

- リポに `.github/PULL_REQUEST_TEMPLATE.md` があれば**骨格として尊重**する。
  個人リポでテンプレに無駄なセクションがあれば削る/埋め草を入れない。
  業務・チームリポの**必須セクション(チェックリスト等)は残す**。迷えばテンプレ
  尊重に倒す。
- Claude Code の PR 末尾トレーラ `🤖 Generated with Claude Code` は**保持**する。

### Compact 版(小規模 PR / 1〜2 概念 / ~200 行未満)

見出しを置かず、地の文 + 必要なら短い箇条書き。

~~~markdown
<1〜2行で「何を・なぜ」。結論先出し>

- <主な変更を概念単位で 1〜3 行>

検証: <テスト/手動確認/未検証 を1行>

Closes #<issue>   <!-- 関連 issue があれば -->
~~~

各部の役割:
- 1行目: BLUF。タイトルと重複させず、結論を立てる。
- 箇条書き: 変更を概念で。ファイル列挙はしない。
- 検証: 1行。動作確認の事実。

### Standard 版(中〜大規模 / 複数概念 / レビュー観点が要る)

~~~markdown
## 概要
<1〜2行で「何を・なぜ」。結論先出し。ここだけで本質が取れること>

## なぜ
<2〜4行。解決する問題/背景。自明なら "概要" に畳んで本節を省く>
Closes #<issue>

## 主な変更
- <概念A: 何をしたか 1行>
- <概念B: 何をしたか 1行>
- <概念C: 何をしたか 1行>

## 検証
- <テスト追加 / 手動手順 / スクショ / 未検証 を簡潔に>

<!-- 任意。必要なときだけ -->
## レビュー観点
- <重点的に見てほしい箇所・トレードオフ・未解決の判断点>

<details><summary>詳細(設計判断 / ログ / 全ファイル)</summary>

<長い内容はここに畳む。デフォルト表示を汚さない>

</details>
~~~

## アンチパターン除去ルール(AI が混入しがちな NG と言い換え)

### A. 冗長な前置き・定型句(削除 or 短縮)

| NG(AIっぽい) | 言い換え / 対処 |
|---|---|
| 「このプルリクエストでは〜を実装します」 | 主語の自己言及を削除し、結論から書く |
| 「〜するために(In order to)」 | 「〜するため / 〜に」(短縮) |
| 「〜という点に注意が必要です(It's worth noting)」 | 重要なら本文へ、そうでなければ削除 |
| 「ご存知の通り / 周知のように」 | 全削除 |
| 「最終的に / 結論として / つまり」を文頭多用 | その文を冒頭(BLUF)へ繰り上げて前置きごと削除 |

### B. 誇張形容・空疎語(具体に置換 or 削除)

- **NG 語**: 「堅牢な(robust)」「シームレスな(seamless)」「強力な」「大幅に改善」
  「劇的に」「包括的な」「最適化された」「活用する(leverage/utilize)」。
- **ルール**: 形容で価値を主張せず、**測れる事実**で示す(「大幅に高速化」→
  「N回の DB 往復を1回に」)。`utilize→使う` `leverage→使う` のように平易語へ。
  形容詞が事実に置換できないなら削除。

### C. 構文上のクセ

- **「Xではなく、Yです(It's not X, it's Y)」構文**: 肯定文に直す(「Yする」)。
- **3連の列挙(rule of three)**: 「象徴し、反映し、示す」式の飾り3点セットは、
  具体1点に削る。
- **自己問答**: 「なぜこうするのか? それは〜だからです」→ 問いを消して答えだけ書く。

### D. 構造の肥大

- **自明な要約の繰り返し**: 「主な変更」で述べたことを「まとめ」で再掲しない。
  要約は冒頭1回だけ(BLUF)。
- **箇条書き過多 / 見出し乱立**: 全文が入れ子箇条書きなのは走査性を逆に下げる。
  概念で束ねて項目数を7以下、見出しを4以下に。
- **diff の逐語再記述**: 「`foo.ts` の42行目を変更し関数 `bar` を追加」式の HOW
  列挙は削除(diff が語る)。body は概念と動機に徹する。
- **絵文字・装飾の過剰**: 見出しごとの絵文字や `🚀✨` は速読に寄与しないので原則
  使わない(Claude Code 末尾トレーラの絵文字は例外として保持)。

### E. メタルール(全文に問う)

- **「事実か。短いか。diff に無いか。」** を全文に問う。3つ全部 yes でなければ
  削るか書き換える。
- **長さの誘惑に抗う**: 「説明が足りないより足りなさすぎる方がレビュアに優しい」。
  不足はレビュアが安く質問できるが、過剰は全員のコストになる。

## few-shot(Before → After)

### 例1: 小規模バグ修正(→ Compact 版に着地)

**Before(AIっぽい冗長な PR body)**

~~~markdown
## 概要
このプルリクエストでは、セッションのクリーンアップ処理に存在していた競合状態
(レースコンディション)の問題を解決するための、堅牢かつ包括的な修正を実装します。
ご存知の通り、この問題は高負荷時において深刻な影響を及ぼす可能性がありました。

## 変更内容
- `session.ts` の `cleanup()` 関数を修正しました
- ミューテックスを追加することで、同時アクセスを防止するようにしました
- これにより、システム全体の安定性が大幅に向上します

## まとめ
本修正により、セッションクリーンアップの競合状態が解消され、より堅牢で
信頼性の高いシステムを実現できるようになりました。
~~~

**After(速読構成 / Compact)**

~~~markdown
セッション cleanup の競合を直し、高負荷時に出ていた 502 を解消する。

- `cleanup()` を mutex で直列化し、同時実行時の二重解放を防止

検証: 既存の負荷再現テストで 502 が出ないことを確認(50 並列 / 1000 req)
Closes #214
~~~

落とした要素: 自己言及の前置き / 「堅牢・包括的」等の誇張 / 「ご存知の通り」/
冒頭と重複する「まとめ」/ 自明な HOW 列挙(diff が語る)。残したのは結論・概念
1点・検証事実。

### 例2: 中規模リファクタ(→ Standard 版に着地)

**Before(AIっぽい)**

~~~markdown
## 概要
本プルリクエストは、認証まわりのコードベースを最適化し、よりクリーンで保守性が高く、
拡張性に優れたアーキテクチャへとリファクタリングする、大規模かつ重要な変更です。

## 詳細な変更点
- `auth/` ディレクトリ配下の全ファイルをリファクタリングしました
- `login.ts` の 30 行目から 80 行目を書き換えました
- `token.ts` に新しい関数 `refreshToken()` を追加しました
- `middleware.ts` のインポート文を更新しました
- 型定義を `types.ts` に移動しました
- テストファイルも併せて更新しました
...(以下、ファイルごとに延々と続く)

このリファクタリングにより、認証システムは飛躍的に改善され、今後の機能追加が
シームレスに行えるようになることが期待されます。
~~~

**After(速読構成 / Standard)**

~~~markdown
## 概要
認証ロジックを「トークン発行」「検証ミドルウェア」「型定義」の3層に分離する。
散在していた責務を集約し、今後のプロバイダ追加を1ファイル変更で済むようにする。

## なぜ
ログイン/トークン更新/検証が `login.ts` に密結合し、変更のたびに全体を読む必要が
あった。プロバイダ追加(#188)に着手する前提として責務分離が必要。

## 主な変更
- トークン発行・更新を `token.ts` に集約(`refreshToken()` 追加)
- 検証を `middleware.ts` に分離、`auth/` の相互依存を解消
- 共有型を `types.ts` へ集約

## 検証
- 既存の認証テスト全パス。ログイン/更新/失効の手動確認済み

## レビュー観点
- `refreshToken()` のリトライ境界(`token.ts`)が主な判断点。それ以外は機械的な移設

<details><summary>移設したファイルの対応表</summary>

| 旧 | 新 |
|---|---|
| login.ts:30-80(発行) | token.ts |
| login.ts:90-120(検証) | middleware.ts |
| 各所の型 | types.ts |

</details>
~~~

落とした要素: 「最適化・クリーン・拡張性に優れた・飛躍的・シームレス」の誇張連発 /
行番号単位の HOW 列挙(→概念に束ね、対応表は折りたたみへ) / 末尾の期待表明。
加えた価値: 動機(なぜ今やるか)/ レビュー観点(機械的移設と判断点の切り分け =
視線誘導)。
