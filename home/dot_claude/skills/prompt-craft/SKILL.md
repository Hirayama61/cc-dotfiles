# prompt-craft — 依頼整形と skill/agent 定義の品質基準

プロンプトを書く負担と「最適な文章になっているか分からない」不安を、
構造テンプレートとチェックリストで解消する。モードは次の優先規則で判別する:

1. 対象が `SKILL.md` / `agents/*.md` の生成・添削なら**定義モード(モード 2)**
   (依頼文の形で来ても定義が対象なら定義モードを優先する)。
2. それ以外の依頼文・箇条書きは**整形モード(モード 1)**。
3. 入力が空・判別不能なら AskUserQuestion でどちらかを確認する。

## モード 1: 依頼整形

雑な依頼(箇条書き・思いつきの羅列)を次の構造に仕上げる:

```text
## 目的(1 文。何が達成されたら成功か)
## 背景(なぜ今それが要るか。既知の制約・過去の経緯)
## 制約(壊してはいけないもの・触ってはいけない範囲・技術制約)
## 成果物(何がどこにできるか。ファイル/PR/レポート)
## 完了基準(検証可能な条件。「動くこと」ではなく「どう確認するか」)
```

手順:

1. 元の依頼文から各節を埋め、**推測で補った箇所に `(推測)` を付けて**草案を提示する。
2. 空欄・曖昧な節だけを AskUserQuestion で確認する(1 問ずつ。全節を尋問しない —
   埋まっている節は確認しない)。**確認で回答が得られない節は `(未確定)` と明記して
   先へ進む**(空欄で止めない)。
3. 確定した構造化プロンプトを提示する。「このまま実行」を選んだ場合は**本 skill を終了し、
   確定プロンプトをメイン会話の依頼として扱う**(実行は skill の責務外)。「文面だけ受け取る」
   なら文面を返して終える。

整形の作法:

- 元の依頼の語彙を保つ(勝手な言い換え・専門語化をしない)。
- 「〜など」「いい感じに」を残さない — 範囲が閉じるまで具体化するか、明示的に
  「範囲は Claude の裁量」と書く。
- 完了基準は必ず検証手順で書く(例: 「テスト X が通る」「画面 Y で Z が表示される」)。

## モード 2: skill / agent 定義の生成・添削

対象: `SKILL.md`(skill)/ `agents/*.md`(agent 型)。

**前段(生成・添削で共通)**: 既存の見本を Read して慣例(頻出の節構成・報告テーブル様式)に
合わせる。skill は `~/.claude/skills/` から 2-3 例、agent は `~/.claude/agents/` から 2-3 例を
Read する。

### 生成の手順

1. 責務・トリガ・必要ツールを聞き取る(不足している項目のみ 1 問ずつ)。
2. 見本を Read する(前段のとおり skill/agent 別に 2-3 例)。
3. 草案を作成し、下のチェックリストで自己照合する。
4. 草案の**全文を提示して人間の承認を得る**。
5. 承認後に Write で配置する(**承認前に書き込まない**)。

### チェックリスト(skill 用)

- [ ] `name` は kebab-case で責務が読める(汎用語 1 語にしない)。
- [ ] `description` に**自然文トリガを含める**(「〜して」と `/名前`。router は description で
  発火判定するため、書いていない起動語では発火しない)。`/名前` の要件は **skill のみ**。
- [ ] `allowed-tools` は最小集合(レビュー系に Write を与えない、等。ツールが権限の実体)。
- [ ] 冒頭 2-3 行で「何を・なぜ」が完結する(本文を全部読まないと責務が分からない定義は失格)。
- [ ] 手順は番号付きで、各手順に**失敗時の分岐**(fail-open / 中断 / 人間確認)を明示する。
- [ ] 守らせたい不変条件は「不変条件」節に分離する(手順の中に埋めると読み落とされる)。
- [ ] 例(入出力・報告フォーマット)を最低 1 つ入れる。抽象指示だけの定義は挙動が安定しない。
- [ ] 他 skill/lib と重複する手順は書かずに参照する(単一情報源。インライン複製は drift する)。
- [ ] 手順の詳細が長くなるなら `reference/`・`scripts/` へ追い出す。

### チェックリスト(agent 用)

- [ ] `name` は kebab-case で責務が読める。
- [ ] `tools` フィールドを**必ず明示**する(省略すると全ツール継承になる)。
- [ ] `model` / `effort` を用途に合わせて指定する。
- [ ] 本文で自己完結させる。**`reference/` 分割は不可**(agent 本文はシステムプロンプトに
  全量注入されるため、外部参照は解決されない)。
- [ ] 手順・不変条件・例の要件は skill 用チェックリストに準じる。

### 行数の目安

- **新規生成は 100 行を目安**に収める。超えるなら手順の詳細を `reference/`・`scripts/`
  (agent は本文内)へ追い出す。
- **既存の添削では行数超過は任意扱い**(既存の慣例に合わせる方を優先する)。

### 添削の出力

指摘を「**必須/推奨/任意**」で列挙し、変更箇所を **unified diff** で示す(全文は求められた
時のみ出す)。

## 連携

- `evolve`(ローカル進化の候補生成)は、候補 skill の品質をこのモード 2 の
  チェックリストで整える(evolve 側から参照される想定)。evolve は cc-dotfiles の
  `feat/local-evolution`(PR #68)で導入。**未導入環境では本 skill 単体で完結する**。
- 反復利用するプロンプトを評価データで最適化したい場合は `empirical-prompt-tuning`
  (ext-skill)を案内する(責務外)。**ext-skills 未導入なら案内のみ**にとどめる。
