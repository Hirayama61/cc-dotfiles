# guide-capture

`~/obsidian/brain/Guides` の **生きたガイド**(context ごとの運用知 current-state(現在状態))へ、
日々の作業で得た学びを蒸留して書き込む。狙いは「Claude Code を使うだけで運用知が
最新状態で蓄積・整理される」こと。

**このガイドは current-state のみ。** 「いつ何をした」の履歴ではなく「今こうする /
こう気をつける」の運用知を書く。「なぜ X を選んだか(履歴・トレードオフ)」は
Decisions の領分で、こことは**直交**する(混ぜない)。

**意味判断は Claude、最終ゲートは人間。** 蒸留・prune・配置(ルート/サブ doc)の提案は
Claude がするが、**書込前に必ず diff を見せ、人間の承認を得てから書く**
(陳腐削除・昇格降格も承認時に行う)。
無検証の自動書込はしない(noise/誤情報が「最新のみ・正しい」を壊す)。

このスキルは 2 経路で呼ばれる:
- **随時 proactive 提案**(作業中に guide-worthy な運用知を学んだら、main がフル
  コンテキストで反映を提案する。CLAUDE.md の規律で発火)。
- **ユーザーの明示起動**(`/guide-capture`、「ガイドに書いて」等)。

---

## 1. 対象 context を解決する

書き込む先のガイドは **context 単位**(通常は repo、横断知はトピック)。

```sh
~/.claude/hooks/lib/resolve-repo-key.sh "$PWD"
```

- 返り値が **repo キー**(例 `dotfiles` / `cc-dotfiles`)なら → `Guides/<repo>/`。
  - worktree 隔離下でもこのリゾルバはメイン clone の repo 名を返す(branch leaf に
    ならない)。だから worktree で作業していても context は repo に集約される。
- **横断知**(特定 repo に紐づかない platform/tool/language の運用知。例 macOS の
  挙動、tmux/fzf、Go の落とし穴)→ `Guides/_topics/`。repo か横断かは学びの性質で
  判断する(「この repo でだけ効く」なら repo、「どの repo でも効く」なら `_topics`)。
- リゾルバが **空文字**を返す(非 git 文脈)時は、学びが横断知なら `_topics`、
  それも曖昧なら**人間に context を確認**してから進む(誤った場所に書かない)。

## 2. ルートガイドを lazy 確保する

context のルートガイドは決定的命名:

- repo: `~/obsidian/brain/Guides/<context>/<context>-ガイド.md`
- 横断: `~/obsidian/brain/Guides/_topics/<topic>.md`

手順:

1. Read を試みる。**在れば**全文を読み、現在状態を把握する。
2. **無ければ**最小構造で新規作成する(下記 §9 の雛形)。親ディレクトリは
   `mkdir -p ~/obsidian/brain/Guides/<context>` で生やす(init では先掘りしない)。
   中身は空でよい — キャプチャで埋まる。新規作成自体も §6 のゲート対象
   (「新しいガイドを作ります」と提示して承認を得る)。

> [!important] ルートは常時規範 + インデックス(ハイブリッド構造)
> SessionStart hook は**ルートガイドだけ**を毎回注入する(サブ doc は注入しない)。
> ルートは **常時規範**(注入されているだけで効かせたい少数の知見)と
> **トリガ付きインデックス行**(いつサブ doc を引くかの信号)の 2 部で構成する。
> 知見の実体は既定でサブ doc に置く(§5)。

## 3. 学びを知見エントリに蒸留する

受け取った学び(テスト規約・実装の注意点・落とし穴など)を、**簡潔な現在状態**へ
蒸留する。**知見 1 項目 = フィールド式知見エントリ**で書き、**既定の置き場所はサブ doc**
(ルートに置くのは §5 の常時規範だけ)。

### フィールド式の型

見出し(H3)を 1 エントリの単位とし、フィールドを箇条書きで持たせる。
**トリガ・規約は必須**、**根拠・NG/OK 例は任意**(書けるときだけ足す)。

```markdown
### 共有 lib は source_hook_lib 経由で読む

- **トリガ**: `hooks/lib/` の lib を source する hook を追加・変更する時。
- **規約**: `source_hook_lib <basename>` を使う。直 source(`. "$LIB"`)は禁止。
- **根拠**: [[2026-06-10-参謀ゲート実装計画]]
- **NG/OK**: NG `if [[ -r "$LIB" ]] && . "$LIB"; then` / OK `source_hook_lib flag-paths || exit 0`
```

- **トリガ**は「どの作業に入った時にこの知見が効くか」を書く(ファイル・コマンド・
  操作で特定できる形に。「気をつける時」のような自己言及は不可)。guide-reviewer が
  「トリガ条件が diff に現れているか」で未遵拠/該当なしを切るので、ここが曖昧だと
  レビューが機械的に効かない。
- **規約**は「今こうする」の断定形で書く。理由の長文はここに書かず、根拠フィールドの
  [[link]] に逃がす。
- H3 見出しが guide-reviewer のチェックリスト 1 行に対応する。見出しは知見の要旨に
  なるよう命名する(「注意点 3」のような無内容な見出しにしない)。

### 書き方の規律

- **冗長な日本語文章を書かない**(Preferences のコーディングスタイルに準拠(`~/obsidian/brain/Preferences/`):
  一文一改行・要点は太字・飾り語/造語なし・専門語はかみ砕く)。
- **履歴形でなく運用知形**で書く。
  - NG(履歴): 「2026-06-09 にテストが落ちたので fixture を追加した」
  - OK(運用知): トリガ「テストを追加する時」/ 規約「fixture は `tests/fixtures/` に置く」
- 該当 context のガイドに既に**同主題のエントリがある**なら、新規追加でなく
  既存エントリの**フィールド更新(置換)**として扱う(§4 の prune と一体)。

## 4. 陳腐・矛盾をチェックする(prune)

**古い情報を残さないことがこのガイドの中核要件。** 蒸留したエントリを置く前に、
対象 doc 内に新情報と**矛盾・重複する古い記述**が無いか走査する。

- 矛盾(古い手順 → 新しい手順に変わった)→ 古い記述の **置換**を提案。
- 重複(同じことが2箇所)→ 1箇所に**集約**し他を**削除**を提案。
- 新情報で**完全に陳腐化**した記述 → **削除**を提案。

prune は §6 の diff に「削除/置換」として必ず含め、**承認時に実行**する
(勝手に消さない)。陳腐記述を「念のため残す」はしない(current-only を壊す)。

## 5. 置き場所を決める(ハイブリッド構造)

**ルート = 常時規範 + トリガ付きインデックス行、知見の実体 = サブ doc**。
新しい知見は**既定でサブ doc 行き**で、ルートに本文を積むのは常時規範だけ
(「肥大してから分割する」はしない — 最初から実体をサブへ置く)。

### サブ doc(既定の置き場所)

- 1 サブ doc = **1 コンテンツ**(1 つの節・機能クラスタ。例「hook 開発の作法」
  「テスト規約」「skills-sync 運用」)。フィールド式知見エントリ(§3)を並べる。
- 置き場所は `Guides/<context>/<topic>.md`。学びが特定 repo を超える**横断知**
  (platform/tool/language)なら `Guides/_topics/<topic>.md`。
- 既存サブ doc の担当コンテンツに収まるなら**そこへ追記**、収まらないなら
  新規サブ doc を作る(新規作成も §6 のゲート対象)。
- サブ doc は**孤立させない** — ルートのインデックス行から必ず [[link]] で繋ぐ
  (ハブ & スポーク。obsidian-memory §7)。横断 `_topics` は複数 repo ルートから
  [[link]] されうるハブノードになる。

### ルートの常時規範(例外的にルート本文へ残す側)

**選定基準は重要度ではなく適用様式** — 「引く判断が期待できない場面で効く」か
「踏むと被害が大きい落とし穴」か。次のどちらかを満たす時だけルート本文に置く:

- **トリガが特定作業に紐づかない**(いつ効くか事前に分からない。例「適用は
  `mise run apply` 経由。`chezmoi apply` を直接呼ばない」)。インデックス行にすると
  「引くべき」と気づけないまま踏む。
- **踏んだ時の被害が大きい**(恒久ブロック・実体破壊・機密漏れなど、後から気づいても
  取り返しがつかない)。読まれなくても注入されているだけで効く価値がある。

満たさないもの(特定の作業に入った時に引けば十分なもの)は**すべてインデックス行**に
落とす。常時規範は SessionStart で毎回注入されるので、**簡潔な箇条書き 1〜2 行**で書く
(フィールド式の展開はしない — トリガが「常時」なのでトリガ行が要らない)。

### トリガ付きインデックス行

ルートのインデックス部の 1 行は **トリガ条件 + 1 行要旨 + [[link]]** で書く。
「いつ引くべきか」の信号を行自体が持つのが要件で、単なる目次にしない。

```markdown
- **hook・lib を書く/変える時** — fail-open・source_hook_lib・bash 3.2 互換の契約 → [[claude-hook開発]]
- **テストを追加/変更する時** — bats 基盤、hermetic 化、characterization の書き換え規律 → [[cc-dotfilesテスト規約]]
```

### 昇格・降格の判断

既存エントリの置き場所は固定ではない。常時規範がインデックス行で足りると分かったら
**降格**(サブ doc へ移す)、サブ doc の知見を繰り返し踏んでいる(引く判断が働いて
いない)と分かったら**昇格**(ルートへ抜き出す)を §6 の diff で提案する。

## 6. diff 提示 → 人間承認 → 書込

**書込前に必ず変更を diff で見せ、承認を得てから書く。** これがこのスキルの安全弁。

- 変更の種類を明示して提示する: **追加 / 置換 / 削除 / 新規サブ doc / 昇格・降格 /
  新規ガイド作成**。サブ doc への追加は**ルートのインデックス行の追加・更新と対で**出す
  (実体だけ増やしてルートから引けない状態にしない)。
- 提示はまとめて出し、人間はスキム確認する(co-author 同様、提示と差分で承認を待つ)。
  重要な二者択一(どの context に書くか、常時規範に上げるか否か等)は `AskUserQuestion` を使い、
  **必ず推奨案を添える**(「僕なら repo 側。理由は…」)。
- 承認後に Edit / Write で書く。**陳腐削除・昇格降格も承認時にまとめて実行**する。
- 承認が得られない/曖昧なら書かない。スコープ外の気づきは申し送りに留める。

## 7. 報告(サイレント禁止)

obsidian-memory §5 と同じく、書いたら必ず明示する:

- `Obsidian: Guides/<context>/<context>-ガイド.md に書き込みました`
- `Obsidian: Guides/<context>/<topic>.md に知見を追加しました(ルートのインデックス行も更新)`
- `Obsidian: Guides/_topics/<topic>.md に書き込みました`

新規作成・置換・削除・昇格降格も同様に何をしたか明示する。

## 8. 規約(obsidian-memory に従う)

ガイドも vault のノードなので obsidian-memory の規約を踏襲する:

- **ファイル名は日本語**(グラフのノードラベルになる)。**bare basename を vault
  全体で一意**に保つ([[wikilink]] は bare 形式 = ベース名のみ)。ルートは
  `<context>-ガイド` 接尾辞で `Projects/<repo>.md` 等と衝突回避。サブ/`_topics` は
  内容が一目で分かる topic 名で一意化(衝突しそうなら接頭辞を足す)。
- **macOS の NFC/NFD**: 日本語ファイル名は NFC に統一する(生成側と参照側=hook の
  [[link]] で同じ文字列を使う。`mv` 後は `python3 -c "import unicodedata"` で検証)。
  詳細は Knowledge `macOS日本語ファイル名のNFC-NFD正規化`。
- **孤立ノートを作らない**: サブ/`_topics` は必ずルート or 関連ノートから [[link]]
  で繋ぐ(§5)。
- **frontmatter 必須**(下記雛形)。`related:` は **bare wikilink**
  (`related: ["[[bare 名]]"]`)。
- 書く前に Grep/Glob で同主題の既存ガイド/サブ doc を確認し、あれば追記(Edit)・
  無ければ新規(Write)。

## 9. 雛形

### ルートガイド(lazy 作成時)

`<context>` を実際のキーに置換して書く。節は骨だけで中身は空でよい(キャプチャで
埋まる)。`related:` には関連 Decisions/Projects を bare wikilink で繋ぐ(無ければ空配列)。

```markdown
---
date: YYYY-MM-DD
tags: [guide, living-guide, <context>]
project: <context>
related: []
---
# <context> 生きたガイド

この repo で作業する際の**運用知の現在状態**。常に最新のみ・自己完結。
「なぜ(履歴・判断)」は Decisions、ここは「今こうする / こう気をつける」の最新だけ。
下の常時規範は読まずとも効く前提で置く。インデックスはトリガに合致した時に [[link]] 先を読む。

## 常時規範

<!-- 引く判断が期待できない場面で効く / 踏むと被害が大きい落とし穴だけをここに置く。
     簡潔な箇条書きで、それ以外はすべてインデックス行へ。 -->

## インデックス(トリガで引く)

<!-- 1 行 = トリガ条件 + 1 行要旨 + [[link]]。実体はリンク先のサブ doc。 -->
```

### サブ doc(`<context>/<topic>.md` / `_topics/<topic>.md`)

1 コンテンツ分の知見をフィールド式知見エントリ(§3)で並べる。
`related:` にルートガイドを入れて孤立を防ぐ。

```markdown
---
date: YYYY-MM-DD
tags: [guide, living-guide, <context>]
project: <context>
related: ["[[<context>-ガイド]]"]
---
# <topic>

<1 行: このドキュメントが担当するコンテンツと、いつ引くか>

### <知見の要旨を見出しにする>

- **トリガ**: <どの作業に入った時に効くか(ファイル・コマンド・操作で特定できる形)>
- **規約**: <今こうする(断定形)>
- **根拠**: [[<関連 Decisions/Knowledge>]]  <!-- 任意 -->
- **NG/OK**: NG `<誤り>` / OK `<正解>`  <!-- 任意 -->
```

横断トピック(`_topics/<topic>.md`)は `project:` を横断トピック名にし、冒頭 1 行を
「複数 repo で共通の運用知」と分かる書き方にする(複数 repo のルートから [[link]]
される前提)。
