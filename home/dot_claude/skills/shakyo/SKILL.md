# shakyo

AI が書いたコードを人間が**写経+編集**して**理解の対価**を払う push 前の手順書。
`self-review`(push 前の品質レビュー隊)の**手前**に立つ姉妹スキルで、「AI 出力の diff を
眺めるだけのセルフレビュー」を「手打ちで再入力しながらスロップを間引き、説明できない設計を
対話で潰す」工程へ前倒しする。

前提は **AI 実装は参照ブランチ・人間は作業ブランチへ写経** の分業:

- **参照ブランチ**: AI がチケットに沿って実装済みの diff を持つブランチ(例 `ai/<topic>`)。
- **作業ブランチ**: 人間が写経して本流へ載せる push 対象。参照ブランチと同じ base から派生し、
  最初は空。人間がここへ手打ちで再入力する。
- 写経対象 = `git diff <base>..参照` の変更。
- 完了後の **照合残差** = `git diff 参照..作業`(作業 = 写経をコミットした作業ブランチの tip)。
  空でない部分 = 人間が意図的に変えた箇所の完全な記録。

理解の対価そのものは消せない(検証コストの保存の変種)。手書き時代はコードを書く行為が
それを強制徴収していたが、AI 生成はその徴収装置を外した。本スキルは支払い箇所を push 前の
ローカル工程に再設置する。

## 責務分界(必読)

このスキルの核は **担当の反転を崩さないこと**。本来 novel な理解(なぜこの設計か)は人間が
own し、機械的な複写と検査は AI が引き受ける。次を必ず守る:

| 主体 | やること | やらないこと |
|---|---|---|
| **AI(本体)** | hunk を機械区分/本質区分に理由付きで区分提案 / 承認された機械区分を作業ブランチへ直接適用 / why 対話の応答役(設計理由・トレードオフ・代替案を説明) / 照合残差を転写ミスと意図的編集に分類 / 残差から AI の悪癖を抽出し guide-capture へ還流提案 / 概要欄の抜け指摘 / 写経時間の記録 | 本質区分を人間の代わりに書かない / PR 概要欄を下書きしない / 区分承認を省いて機械区分を勝手に広げない |
| **人間** | 機械区分の一覧を流し見承認 / 本質区分を手打ちで写経しつつスロップ削除・命名修正・不要変更の間引き / 説明できない hunk で手を止め why を問う / PR 概要欄を自分で書く | 写経を AI に肩代わりさせない |

「機械区分の承認を人間が挟む」のは、区分判定がスロップの新しい通り道になるのを防ぐ安全弁。
allowlist の自動判定(列挙漏れ・境界ケースが穴)ではなく人間承認を挟む設計を採る。

## 手順

### 0. 前提解決

参照ブランチ・作業ブランチ・base を確定し、写経対象 diff を取得する。

- 現在ブランチ(`git branch --show-current`)が **作業ブランチ**である前提。参照ブランチは
  ユーザに確認するか、命名規約(`ai/*` 等)から推定して確認を取る。
- **参照ブランチ名は受領直後に検証してから使う**。`git rev-parse --verify "<参照>^{commit}"`
  で実在する commit へ解決し、以降の git 操作はクォートした変数(検証済み ref / SHA)で扱う。
  参照ブランチは自動生成や remote 由来でメタ文字を含みうるため、未検証名を直接コマンドへ
  展開しない(self-review が untrusted 入力に採る姿勢と揃える)。
- base ref は参照ブランチと作業ブランチの分岐点。`git merge-base "<参照>" HEAD` で決める。
- 写経対象を取得: `git diff <base>..<参照>`(AI の変更全体)+ `git diff --stat <base>..<参照>`
  でファイル一覧と規模を掴む。作業ブランチが空(未着手)であることを確認する。
- **開始時刻を捕捉**: `date +%s` の値を控える(手順 7 の実測に使う)。

### 1. hunk 区分の提案(機械/本質)

写経対象の各 hunk を **機械区分** / **本質区分** に、AI が **理由付きで** 提案する。

- **機械区分**: 読めば一瞬で正しさが判る変更。パス変換・一括置換・export/import の追加のみ・
  生成ファイル・定型ボイラープレートなど。ロジックや設計判断を含まない。
- **本質区分**: ロジック・設計判断を含む変更。写経+編集の対象。
- **迷えば本質区分に倒す(default-substantive)**。機械/本質の境界が曖昧な hunk、副作用が
  読み切れない hunk は本質区分として扱う。区分判定を「楽をする方」に漂わせない安全側の既定。
- 提案はファイル×hunk 単位で、各機械区分に一行の理由(なぜ機械的と判断したか)を添える。

### 2. 機械区分の承認と直接適用

人間は **機械区分とされた一覧だけ** を流し見して承認する。

- `AskUserQuestion` で機械区分の一覧を提示し、`全て承認` / `一部を本質区分へ差し戻す` を問う。
  差し戻された hunk は本質区分に移す(手順 3 で写経)。一覧は流し見できる粒度に保つ
  (件数が多ければファイル単位でまとめて提示)。
- 承認された機械区分を **AI が作業ブランチへ直接適用**する。参照ブランチの当該 hunk を
  `git apply` の部分適用、または Edit で作業ツリーへ転記する(手段は問わない)。
- 機械区分は人間が手打ちしないため、この部分の照合残差は構造的にゼロになる。

### 3. 本質区分の写経(ファイル単位・why 停止)

本質区分を **ファイル単位** で提示し、人間が手打ちで再入力する。ここが理解の対価の本体。

- ファイルごとに、写経すべき本質区分の hunk を提示する。人間は参照を見ながら作業ブランチへ
  **手打ちで再入力**し、その過程でスロップ(無駄コメント・AI 語彙・冗長な抽象)を削除し、
  命名を直し、不要変更を間引く。
- AI は各ファイルで **why が非自明な hunk を先出し**する(なぜこの分岐か、なぜこの依存か等)。
  人間がその why を自分の言葉で説明できるか確認する。
- **説明できない hunk で手を止める**。人間が設計理由を説明できなければ、そのファイルを完了と
  せず **why 対話** に入る。AI は設計理由・トレードオフ・検討した代替案を説明する。写経の打鍵
  では埋まらない「なぜ」を、この対話が埋める(レビュー指摘への即答力はここで作られる)。
- 対話で腑に落ちた上で写経を完了する。理解を作るのは複写ではなく編集と対話の瞬間なので、
  ここを飛ばさない。

### 4. 照合残差の検査(転写ミス vs 意図的編集)

全ファイルの写経が済んだら照合残差を取り、二種に分類する。

- **写経を作業ブランチへコミットしてから** `git diff <参照>..HEAD` で残差を取得する。
  two-dot 比較は 2 コミット間を見て未コミットの作業ツリーを含まないため、写経が未コミットだと
  残差が「AI 変更全体の反転」になり分類が丸ごと壊れる。コミット前に途中確認したいときは
  作業ツリーを対象にする `git diff <参照>`(コミット↔作業ツリー比較)を使う。
- 各残差 hunk を AI が **転写ミス**(意図せぬ差=打ち間違い・写し漏れ)か **意図的編集**
  (スロップ削除・命名修正・不要変更の間引き)に分類提案する。
- **転写ミス**は修正候補を提示し、人間確認の上で参照どおりに復元する(打ち間違いは理解の
  対価ではなくノイズ)。
- **意図的編集**はそのまま記録する。残差が意図的編集のみになれば **転写ミスはゼロ**で、参照と
  一致した(残差ゼロの)部分は AI 版(参照ブランチ)で通った検証結果をそのまま持ち越せる。
  ただし意図的編集を入れた領域(不要変更の間引き・命名一括修正など挙動を変えうる編集)は
  参照版の検証が当てはまらないので、**作業ブランチ側で再検証する**(この分類は AI の提案+人間
  確認であって挙動同等性の証明ではない)。
- 分類と最終残差を人間へ提示して合意を取る。

### 5. 悪癖抽出 → guide-capture 還流

意図的編集は「人間の手直し vs AI 出力」の実データ。ここから AI の反復パターンを抽出する。

- スロップ削除・命名修正の残差に共通する **AI の悪癖**(繰り返し出る無駄コメント、好む冗長
  抽象、誤った命名慣習など)を AI が抽出して提示する。
- 抽出した悪癖を **guide-capture** で外部脳の生きたガイドへ還流するよう提案する(人間ゲート。
  proactive 提案に留め、書込は guide-capture 側の diff 承認で行う)。
- 安定した悪癖は後日 **guide-to-repo** で仕事リポの AGENTS.md へ投影する既存経路に載る
  (本スキルでは投影まではやらない。提案のみ)。

### 6. PR 概要欄(人間が書く / AI は抜け指摘のみ)

概要欄は **人間が自分で書く**。「自力で書けない」こと自体を理解不足の検知器にする。

- AI は概要欄を **下書きしない**。人間が「何をしてほしいか・どこを見るか」を自分の言葉で書く。
  ここで詰まったら理解が足りていない合図なので、手順 3 の why 対話へ戻る。
- 人間が書いた概要に対し、AI は **抜けだけを指摘**する: 検証済み項目の記載漏れ、影響範囲
  (触れた領域・想定される副作用)の記載漏れなど。文面の代筆はしない。

### 7. 写経時間の記録と縮退判断

写経時間を実測して記録し、縮退基準の判断材料にする。

- 手順 0 の開始時刻から経過分を算出し(`date +%s` の差 / 60)、人間へ提示する。
- repo キーを `~/.claude/hooks/lib/resolve-repo-key.sh "$PWD"` で導出し、
  `~/obsidian/brain/Tasks/<repo>/写経時間ログ.md` へ「日付 / ブランチ / 変更ファイル数 /
  経過分 / 縮退の要否」を 1 行追記する(doc-gravity。書込は diff 提示 → 人間承認のゲート付き)。
  縮退基準はPR を跨いで判断するため、その場報告で消さず永続化する。
- 事前に決めた縮退基準 N(1 PR あたり N 分)を超えたら、「次回から核心ファイルのみ写経へ縮退」を
  提案する。形骸化(手だけ動く自動運転)する前に設計側が折れられるようにする。

## 原則

- **担当の反転を崩さない**。本質区分の写経と PR 概要欄は人間が own し、AI は肩代わりしない。
  肩代わりした瞬間に理解の対価の徴収装置が再び外れる。
- **機械区分の承認は省かない**。区分判定はスロップの新しい通り道になりうるので、AI の理由付き
  提案に対し人間の流し見承認を必ず挟み、迷えば本質区分に倒す(default-substantive)。
- **照合残差は二役**。転写ミスゼロの確認(参照と一致した部分の検証結果を持ち越す根拠。意図的
  編集を入れた領域は再検証)と、AI の悪癖の実データ(guide への還流源)を兼ねる。だから残差を
  空にしきらず、意図的編集は記録に残す。
- `self-review` との関係: shakyo は理解の対価を払う工程で **self-review の手前**に立つ。
  写経で作った作業ブランチを push する前に self-review で品質レビューを通す(責務は別)。
- **未検証の前提**: 10 ファイル規模 PR の写経時間コストが持続可能かは未検証。手順 7 の実測で
  運用しながら検証し、超過が続けば縮退する。
- **強制力のギャップ(現状の位置づけ)**: 本スキルは手順書であり、shakyo 単体の push 強制 hook は
  まだ無い。強制力は self-review の既存 push ゲートに相乗りしている(写経で作った作業ブランチも
  push 前に self-review を通す運用)。shakyo 自体の強制 hook 化は follow-up とする。
