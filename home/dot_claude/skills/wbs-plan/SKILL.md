# wbs-plan

案件の要件とデザインから、機能が網羅された WBS と課題を facts-only の JSON に作る. 実装前に課題を
潰し切り、共通化を事前判断するための成果物を生成する. **PR 作成・レビュー・チケット起票は本スキルの
責務外**(完成 JSON を各案件の実装ワークフローに渡す).

## 成果物

- `~/obsidian/brain/Tasks/<repo>/<案件名 or チケットID>-wbs.json` を1案件1ファイルで生成・更新する.
- スキーマと初期テンプレは同梱の `reference/wbs-schema.jsonc` に従う(Claude はスキル基準ディレクトリからの
  相対で Read する). この JSON が単一の正.
- 人間は同梱の `~/.claude/skills/wbs-plan/reference/wbs_viewer.html` をブラウザで開き、右上「JSON を開く」で
  当該 JSON を選んでダッシュボード閲覧する(read-only). Claude は JSON を直接 Read/Write/Edit する.
- 階層は 案件 → 画面・ストーリー → 機能 → {FE タスク, BE タスク}. 機能配下に FE/BE を独立タスクとして
  持たせ、各々に独立の status と pr を持つ. BE 作業の無い機能は tasks に BE を入れない.

## facts-only 原則

- 計算・派生できる値は JSON に保存しない. タスクが持つのは 領域 / status / pr だけ.
- status 語彙(進捗の事実): 未着手 / 実装中 / PR化済 / レビュー中 / マージ済.
- ブロックは保存しない=派生. あるタスクは、それを ブロック対象 に含む課題が確認先=人間系(企画/デザイナー/BE)
  かつ status が 解決 でない(open / 確認中)のとき「ブロック中」とみなす(viewer が赤表示). スキルも同じ規則で判定する.
- 課題リンク・共通化リンクをタスクに重複保存しない. 課題の ブロック対象(タスクキー <機能id>.<領域>)と
  共通化候補の 対象機能(機能ID)が単一の源で、表示側が逆引きする.

## 課題のブロック規約

- 確認先が 企画 / デザイナー / BE の課題は人間しか解けない=ハードブロック. status が 解決 になるまで
  (open も 確認中 も)当該タスクを実装しない. 確認に出した途中(確認中)でも、回答が確定する 解決 まで止める.
- 確認先が 自己解決 の課題は、生成中にコード/仕様を読んでその場で解決し status: 解決 + 解決メモ.
- 課題 status 語彙: open / 確認中 / 解決.

## 手順

### Phase 0: 入力確定

会話と $ARGUMENTS から以下を確定する. 不足は AskUserQuestion で1問ずつ確認する.

- 要件の参照先(Confluence URL / ローカル仕様書パス 等)
- デザインの参照先(Figma URL / ローカルパス 等)
- 対象リポジトリ(<repo>. JSON の置き場所と既存コード突合に使う). <repo> は doc-gravity 規約に従い
  `bash ~/.claude/hooks/lib/resolve-repo-key.sh <対象案件リポのパス>`(省略時は cwd)で既定値を導出し、
  それを提示して確認する. **リゾルバは解決不能でも空文字を stdout に返し exit 0 する(失敗しない)ので、
  exit status ではなく stdout が空かどうかで判定する. 空なら未解決の合図として AskUserQuestion で確認する.**
  cwd と別リポを対象にする場合も人間に確認する.
- 案件名 or チケットID(JSON ファイル名に使う)

要件源/デザイン源が認証付き URL(Figma / Confluence 等)の場合、WebFetch は失敗しうる. 失敗したら人間に
エクスポート/該当本文/スクショ/ローカルファイルのいずれかを求め、曖昧なまま分解に進まない.

~/obsidian/brain/Tasks/<repo>/<案件>-wbs.json が既にあれば Read し、途中から再開する(JSON 自体が
進捗の state. 私有の進捗ファイルは作らない).

### Phase 1: 着手順の整理

既存チケットの有無を AskUserQuestion で確認する.

- チケットあり → そのチケット/ストーリー順で画面・ストーリーを列挙する.
- チケットなし → 要件とデザインから画面・ストーリー一覧を作り、依存関係から着手順を提案する.

reference/wbs-schema.jsonc の形で JSON の骨格(案件メタ + 画面[] の id/name/順序、機能は空配列)を
Write し、着手順を人間に提示して確認を取る.

### Phase 2: 画面ごと逐次分解(本体)

着手順の先頭から 1画面ずつ 処理する. 各画面で:

1. 要件・デザイン・既存実装コードを突合する. 既存コードの大量読み・突合は Agent(scout か delegate)に
   委譲し、結論(機能候補・既存実装との差分・デザインの不整合)だけ受け取る.
2. 画面を機能(葉)に分解し、各機能に FE タスク / BE タスクを起こす(BE 作業が無ければ BE を入れない).
3. 次を課題として抽出し Q-NN で採番、種別(仕様不明 / デザインミス / 既存実装乖離 / 技術検討)と
   確認先(企画 / デザイナー / BE / 自己解決)を判定する.
   - 自己解決可 → その場で解決し status: 解決 + 解決メモ.
   - 人間系 → status: open. ブロック対象 に当該タスクキーを入れる(ブロックは派生なのでタスク側はいじらない).
4. JSON の当該画面・課題を Edit/Write で更新保存する.
5. 画面末チェックポイント: この画面で出た人間系 未解決(open / 確認中)課題を AskUserQuestion でまとめて提示し、
   確認先・優先度を人間にトリアージさせる. その後 次の画面へ.

全画面を処理し終えるまで繰り返す.

### Phase 3: 共通化パス(全画面分解後)

全機能を横断し被っているロジックをクラスタ化する. 各クラスタを 共通化候補 に C-NN で起こし
(共有ロジック・対象機能・推奨)、該当機能IDを 対象機能 に結ぶ. やる/やらないの最終判断は人間に
握らせる(AskUserQuestion で提示し 判断 を 採用/見送り に更新).

### Phase 4: ハンドオフ

- 完成 JSON のパスと、`~/.claude/skills/wbs-plan/reference/wbs_viewer.html` をブラウザで開いて JSON を
  選ぶ手順を提示する(人間が辿れるよう絶対パスで示す).
- 未解決(open / 確認中)の人間系課題が残っていれば一覧化し、「これらが解決するまで対応タスクは着手不可」と明示する.
- 実装はこのスキルの責務外であることを伝え、各案件の実装ワークフローに JSON を渡すよう促す.

## 更新規約(実装セッションも従う)

- 変更後は必ず 更新.at(`date -Iseconds` 等で取得)と 更新.by を書き換える.
- 実装が進んだらタスクの status を 未着手→実装中→PR化済→レビュー中→マージ済 と進め、pr に PR 番号を入れる.
- 課題を確認に出したら課題の status を 確認中 にする. 人間系は 確認中 でも 解決 までブロックが続く.
  回答が確定したら status を 解決 にし 解決メモ を書く(派生でブロックが外れる).
- JSON に無い機能は実装しない(網羅前提). 新たに必要と判明したら、まず機能を JSON に追記してから実装する.

## 原則

- 実装しない / PR を作らない / チケットを起票しない. 本スキルは WBS と課題の生成・維持だけ.
- JSON が単一の正. 進捗 state はこのファイルから常時再計算し、私有 state を持たない.
- facts-only. ブロック・課題逆引き・共通化逆引き・集計は保存せず派生で出す.
- 大量読みは委譲. 既存コード突合は scout/delegate に出し結論だけ受け取る.
- 網羅前提. JSON に全機能を載せ、載っていない機能は実装させない.
- 重い判断は人間へ. 課題のトリアージと共通化の採否は AskUserQuestion で握らせる.
