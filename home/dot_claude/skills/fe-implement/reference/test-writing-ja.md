# テスト日本語の規範(fe-implement 手順 4 が必参照)

テストの価値は「落ちた時に、何が壊れたかテスト名だけで分かる」こと。
観点とその日本語が読めないテストは、レビューで業務を止める(実害の観測済み)。

## 観点の立て方

- **1 テスト 1 観点**。「正しく動作すること」のような複合・曖昧観点を作らない。
- **仕様の語彙をそのまま使う**。Confluence・画面の文言にある言葉でテスト名を書く。
  自分で言い換えたり、新しい用語を発明したりしない(造語禁止)。
- 観点の構成は「状況 → 操作 → 期待結果」。期待結果が書けない観点はテストではない。

## describe / it の書き方

- `describe` = 対象(コンポーネント名・機能名)。`it` = 「〜の場合、〜すると、〜になる」の
  形に収まる具体文。
- 「〜が正しいこと」「〜を確認する」で終わらせない — 何がどうなれば正しいのかを書く。

## 良い例 / 悪い例

| ✗ 悪い例 | ✓ 良い例 | なぜ |
|---|---|---|
| `it('検索が正しく動作すること')` | `it('検索語に一致する行だけが一覧に表示される')` | 「正しく」は期待結果を言っていない |
| `it('バリデーションを確認する')` | `it('メールアドレスが空のとき送信ボタンが押せずエラー文言「メールアドレスを入力してください」が出る')` | 確認する、はテストの動作であって観点ではない |
| `it('エッジケースのハンドリング')` | `it('検索結果が 0 件のとき「該当するデータがありません」が表示される')` | 抽象語(エッジケース・ハンドリング)は仕様の語彙でない |
| `it('ユーザーインタラクションの整合性が担保される')` | `it('行をクリックすると詳細画面に遷移する')` | 雰囲気語の連結は観点が存在しない兆候 |
| `describe('正常系/異常系')` だけの階層 | `describe('受注一覧の検索フィルタ')` | 正常/異常の二分は観点の分類にならない。機能で括る |
