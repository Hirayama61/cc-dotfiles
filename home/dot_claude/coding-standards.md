# コーディング規約

## コメント方針

- **既定はコメントなし**。コードと命名で意図を表現する。
  書く前に「これはコードから読み取れないか」を自問する。
- 書いてよいのは次の2種だけ:
  - **workaround**: 苦肉の策。原因と参照先(issue/PR/URL)を添える。
  - **非自明な前提・制約**: 外部仕様、順序依存、既知の落とし穴。
- **経緯・採否理由・設計判断の背景はコードに書かない**。
  Decisions ノート(外部脳)・コミットメッセージ・PR 説明に書く。
- 変更理由の説明・レビューア向けの正当化・コードの言い換えはコメントにしない。
- 周辺がコメント過多でも倣わない。触った箇所の不要コメントは削ってよい。
